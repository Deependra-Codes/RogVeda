# Docs Lifecycle Pattern

Status: active
Date: 2026-04-22

## Core Rule

No important architecture or behavior claim should exist without a traceable path back to intent, research, spec, or verification.

## Lifecycle

1. `Intent`
   what problem we are solving and why now
2. `Research`
   options, tradeoffs, risks, and recommendation
3. `Spec`
   required behavior and acceptance shape
4. `Implementation`
   the smallest correct change in the right boundary
5. `Validation`
   what was checked, what passed, what remains uncertain

## Quality Rules

- prefer primary or official sources
- separate evidence from inference
- keep records short but decision-useful
- update indexes when new durable docs are added
